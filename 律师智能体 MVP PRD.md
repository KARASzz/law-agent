# 律师智能体 MVP PRD

## 1. 文档信息

### 1.1 产品名称

律师智能体 MVP

### 1.2 版本

PRD v1.0

### 1.3 目标上线周期

6-8 周

### 1.4 核心目标

在飞书中交付一个律师愿意每天使用的最小可用产品，先解决三个高频场景：

1. 法规速查
2. 类案检索
3. 文书初稿生成

---

## 2. 产品背景

律师真正高频、刚需、可验证的任务，不是"哲学级认知建模"，而是：

- 快速找法条
- 快速找案例
- 快速拉出文书骨架

法律 AI 近两年的头部产品也都围绕这些工作流展开，而不是空泛聊天：  
vLex 强调可信来源驱动的研究、文档分析和诉讼工作流；Harvey 强调统一平台、知识、文档、工作流代理与安全边界。[<sup>5][</sup>6]

---

## 3. 产品目标与非目标

## 3.1 产品目标

### O1 提高律师检索效率

把一次法规/案例查询从 10-30 分钟压缩到 1-3 分钟

### O2 提高文书起草起点质量

把起诉状初稿生成时间从 1-2 小时压缩到 10-20 分钟

### O3 建立信任

让律师相信：

- 答案有来源
- 结论可复查
- 文书不会乱导出
- 系统不会乱学坏

## 3.2 非目标

- 不做自动判决预测
- 不做自动对外提交
- 不做合同全生命周期系统
- 不做多 Agent 协作可视化
- 不做高度自治提醒系统

---

## 4. 目标用户

## 4.1 核心用户

### 用户 A：青年律师

- 经常做检索
- 经常起草基础文书
- 需要提高交付速度

### 用户 B：律师助理/实习律师

- 负责材料整理和初稿搭建
- 需要标准化结构辅助

### 用户 C：团队负责人

- 关注文书底稿质量
- 关注团队统一口径

## 5. 典型使用场景

## 5.1 场景一：法规速查

用户在飞书私聊发送：  
"公司拖欠工资三个月，员工是否可以立即解除劳动合同并要求补偿？"

系统返回：

- 结论摘要
- 对应法条
- 生效状态
- 适用说明
- 风险提示

## 5.2 场景二：类案检索

用户在飞书群里 `@机器人`：  
"帮我找建设工程领域关于实际施工人主张工程价款的相关案例"

系统返回：

- 相关案例 3-5 个
- 裁判要旨
- 地域/法院层级
- 参考价值说明
- "案例仅供参考"提示[^4]

## 5.3 场景三：文书初稿

用户上传事实摘要和当事人信息，输入：  
"生成民事起诉状初稿"

系统返回：

- 文书结构化初稿
- 缺失材料提示
- 法律依据建议
- 导出前强制确认

## 6. MVP 功能清单

## 6.1 F1 法规速查

### 功能描述

根据自然语言问题，在法律知识库中检索相关规范文件并生成带来源的回答。

### 输入

- 自然语言问题
- 可选法域
- 可选时间范围

### 输出

- 结论
- 法条原文摘要
- 原始来源
- 生效状态
- 置信度
- 风险提示

### MVP 范围

- 支持全国法域
- 支持基础法条与司法解释
- 不支持复杂跨法域冲突分析

### 验收标准

- 法条引用准确率 >= 98%
- 平均响应时长 <= 15 秒
- 无来源输出率 <= 1%

---

## 6.2 F2 类案检索

### 功能描述

根据案由、争点或事实描述，返回高相关案例列表和裁判要旨。

### 输入

- 自然语言案情/争点描述
- 可选案由
- 可选地域

### 输出

- 案例标题
- 审理法院
- 裁判日期
- 裁判要旨
- 相关法条
- 来源链接

### MVP 范围

- 每次返回 3-5 个案例
- 不做胜诉概率预测
- 不做自动策略建议结论化输出

### 验收标准

- Top3 相关性满意率 >= 75%
- 案例真实性 = 100%
- 裁判要旨抽取人工可接受率 >= 85%

---

## 6.3 F3 文书初稿生成

### 功能描述

根据案件事实和模板规则生成标准结构的民事起诉状初稿。

### 输入

- 原告信息
- 被告信息
- 事实与理由
- 诉讼请求
- 证据目录（可选）

### 输出

- 标准格式起诉状初稿
- 待补材料清单
- 可能遗漏项提示
- 法律依据建议

### MVP 范围

- 只支持"民事起诉状"
- 不支持自动提交法院
- 不支持自动盖章
- 不支持对外直接发送

### 强制规则

- 用户必须点击"确认已审阅"后，才允许导出
- 每次导出写入审计日志

### 验收标准

- 草稿结构完整率 >= 90%
- 用户首次保留率 >= 70%
- 导出前确认率 = 100%

## 7. 功能流程

## 7.1 法规速查流程

```text
用户提问
→ 主Agent识别为法规查询
→ 调用法规检索工具
→ 检索法律/司法解释
→ 调用引用核验工具
→ 生成结论+来源
→ 风险标签工具判级
→ 返回用户
→ 记录采纳/修改行为
```

## 7.2 类案检索流程

```text
用户提问
→ 主Agent识别为类案检索
→ 提取案由/争点/地域
→ 调用案例检索工具
→ 返回候选案例
→ 调用摘要工具抽取裁判要旨
→ 风险提示"案例仅供参考"
→ 返回用户
→ 记录点击与使用情况
```

## 7.3 文书生成流程

```text
用户选择文书生成
→ 收集表单/文本输入
→ 校验必要字段
→ 调用文书模板工具
→ 生成初稿
→ 调用风险标签工具
→ 调用引用建议工具
→ 返回初稿
→ 用户编辑
→ 用户点击"确认已审阅"
→ 导出/复制
→ 写审计日志
```

## 8. 交互设计

## 8.1 飞书私聊

适合：

- 法规问答
- 文书初稿
- 合同单文件分析

### 交互样式

- 文本卡片
- 引用折叠区
- 风险标记
- 操作按钮：

  - "查看依据"
  - "继续追问"
  - "生成文书"
  - "确认导出"

## 8.2 飞书群聊

适合：

- 团队共享检索结果
- 快速类案讨论
- 法规速查

### 规则

- 仅 `@机器人` 响应
- 默认返回摘要版
- 点击后展开详细版

## 8.3 文件上传

- 上传后先不自动大模型全量处理
- 先弹出任务选择：

  - "提取摘要"
  - "找风险点"
  - "生成起诉状材料"
  - "只做检索"

这样能减少误触发和 Token 浪费。

## 9. 数据与权限要求

## 9.1 必须存的数据

- task_id
- session_id
- trace_id
- 用户 ID
- 输入摘要
- 输出摘要
- 工具调用记录
- 结果确认记录
- 导出记录

## 9.2 不默认永久存储的数据

- 原始敏感附件正文
- 全量聊天长期上下文
- 未经确认的中间草稿

## 9.3 权限要求

- 助理不可直接导出正式文书
- 律师可导出本人文书
- 合伙人可查看团队汇总指标
- 管理员不可直接查看正文明文

---

## 10. 非功能需求

## 10.1 性能

- 法规查询 <= 15 秒
- 类案检索 <= 20 秒
- 文书初稿 <= 30 秒

## 10.2 可用性

- 可用率 >= 99%
- 出错时给明确失败原因，不做沉默失败

## 10.3 安全

- 私有化部署
- Token 认证
- 全链路日志
- 敏感字段脱敏
- 可删除会话

## 10.4 审计

每个高风险输出必须可追溯：

- 谁发起
- 何时生成
- 调了哪些来源
- 谁确认导出

这正符合生产级 Agent 对 traceability、decision provenance 和 policy event logging 的要求。[<sup>2][</sup>3]

## 12. MVP 的技术方案

## 12.1 技术栈

- 入口：飞书机器人
- 网关：OpenClaw
- 编排：单主 Agent
- 模型：

  - 简单问答：轻量模型
  - 文书生成：中等推理模型
- 检索：关键词 + 向量混合检索
- 存储：

  - 任务库
  - 审计库
  - 法律知识库
  - 用户偏好快照库

## 12.2 不上什么

- 不上自由能引擎
- 不上复杂多 Agent 框架
- 不上自动主动提醒
- 不上复杂长期记忆漂移机制

## 14. 风险与应对

## 14.1 法律引用错误

- 应对：引用核验工具强制前置[^4]

## 14.2 检索结果不相关

- 应对：关键词 + 向量混检；保留人工筛选入口

## 14.3 文书草稿质量不稳定

- 应对：限制支持文书类型，先做起诉状一个模板

## 14.4 成本超支

- 应对：文件分析先选任务；上下文裁剪；轻量模型优先[<sup>2][</sup>3]

## 14.5 用户不信任

- 应对：固定展示来源、风险、置信度；不抢律师最终决定权[^4]

**先证明工具价值，再引入个性化；先证明可信，再引入更强自治。**
