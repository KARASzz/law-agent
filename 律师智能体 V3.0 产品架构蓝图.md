# 律师智能体 V3.0 产品架构蓝图

## 1. 产品定义

### 1.1 产品名称

- 中文名：律师智能体工作台
- 内部代号：LA-Workbench V3.0
- 产品定位：面向律师执业流的可观测法律智能工作流系统

### 1.2 一句话定义

通过飞书接入、OpenClaw 网关、中心编排 Agent、法律知识治理、风险审查与全链路审计，帮助律师更快完成法规检索、类案分析、文书起草和合同审查，同时保证输出可追溯、可复核、可回滚。[<sup>1][</sup>2][^4]

### 1.3 不做什么

- 不做"替代律师决策"
- 不做"自动对外出具法律意见"
- 不做"无来源结论生成"
- 不做"黑箱长期人格进化"
- 不做"一开始就多 Agent 群体协作"

---

## 2. V3.0 的设计原则

### 2.1 五条总原则

1. **律师主导，AI辅助**
2. **来源先于答案**
3. **审计先于自治**
4. **工作流优先于概念美学**
5. **中心编排优先于 Agent 扩张**

### 2.2 为什么这样设计

多智能体并不是天然更优。Google/MIT 的研究指出，多工具场景下会出现明显的"工具-协调"权衡；当单 Agent 基线已经不低时，继续增加 Agent 往往收益递减，集中式编排反而更稳。[<sup>1]而企业级智能体正在把"可观测性"当成控制平面：不仅要看响应内容，还要能重建用户意图、规划过程、工具调用、数据访问、策略事件和最终结果。[</sup>2][^3]

## 4. 核心模块说明

## 4.1 交互入口层

### 目标

让律师在原有工作场所中完成任务，不强迫迁移到新系统。

### 设计

- 飞书私聊：个人检索、文书初稿、合同审查
- 飞书群聊：团队研判、共享案例、法规速查
- 文件上传：合同、起诉状、证据材料、判决书
- 指令菜单：

  - "/查法规"
  - "/找案例"
  - "/写文书"
  - "/审合同"

### 规则

- 群聊默认只对 `@机器人` 响应
- 文件上传默认不自动分析，先提示用户选择任务类型
- 每个任务生成单独 task_id，便于审计

---

## 4.2 网关与接入层

以 OpenClaw 为接入网关，承担：

- 飞书消息接收与发送
- 用户身份映射
- 会话隔离
- 服务守护进程
- 事件日志入库

### 关键策略

- ​`auth=token`
- ​`bind=loopback`
- 飞书用户白名单
- 群会话 requireMention=true
- 所有任务分配 request_id / session_id / trace_id

### 为什么必须这样

OpenClaw 属于高权限智能体网关，自托管和强认证是底线；法律场景不能接受无认证对外暴露。[<sup>2][</sup>3]

---

## 4.3 中心编排层

### 只保留一个主编排 Agent

主 Agent 负责：

- 判断这是不是法律问题
- 判断属于法规/类案/文书/合同哪一类
- 判断风险等级
- 调用合适工具
- 汇总结果并组织输出

### 为什么不是多 Agent 联席会议

因为法律任务对"稳定、可解释、可回放"的要求高于"想象力"。  
多 Agent 适合探索性任务，法律场景更多是责任性任务。[<sup>1][</sup>4]

### 少量专用子代理

只保留两个：

#### Research Agent

负责：

- 法规检索
- 类案检索
- 引用核验
- 地域/法域过滤

#### Document Agent

负责：

- 文书模板选择
- 草稿生成
- 风格重写
- 结构补全

---

## 4.4 工具能力层

### 工具设计思路

所有能力做成"工具"，而不是一开始做成"人格化 Agent"。

### 工具清单

#### 1) 法规检索工具

输入：

- 问题文本
- 法域
- 时间范围

输出：

- 法条原文
- 生效状态
- 摘要说明
- 来源链接

#### 2) 类案检索工具

输入：

- 争点描述
- 案由
- 地域偏好

输出：

- 3-5 个高相关案例
- 裁判要旨
- 引用来源
- 相关程度评分

#### 3) 引用核验工具

输入：

- 法条编号/案例名/引用文本

输出：

- 是否真实存在
- 是否引用准确
- 是否存在废止/失效/法域不匹配

#### 4) 文书生成工具

输入：

- 事实摘要
- 当事人信息
- 请求事项
- 文书类型

输出：

- 标准结构草稿
- 待补材料清单
- 风险提示

#### 5) 风险标签工具

输入：

- 任务类型
- 输出内容
- 是否涉及对外提交/法律判断/金额/程序期限

输出：

- 低/中/高风险
- 推荐操作

## 5. 上下文系统

## 5.1 四层记忆设计

### 第一层：会话记忆

- 保存当前对话最近若干轮
- 生命周期短
- 仅服务当前问题
- 任务结束后压缩

### 第二层：任务记忆

- 保存某次文书、某份合同、某次类案分析的中间状态
- 和 task_id 绑定
- 可复盘
- 可继续编辑

### 第三层：用户偏好快照

这是你原来"画像"概念的收敛版。

内容包括：

- 输出偏好：简洁/详细/先结论/先论证
- 常用领域：民商、劳动、刑事、知产
- 常用模板：起诉状、律师函、审查意见
- 常见改写点：标题、事实结构、引用方式
- 风险偏好：保守/中性/进取

### 第四层：机构知识记忆

- 律所模板
- 内部审稿规范
- 合伙人常见批注规则
- 过往标准问答
- 固定客户偏好

---

## 5.2 偏好快照机制

### 不做连续人格进化

改成：

- 每周形成一个 snapshot
- snapshot 可审计
- snapshot 可回滚
- snapshot 更新幅度受阈值限制

### 数据结构建议

```json
{
  "user_id": "lawyer_001",
  "snapshot_version": "2026W15",
  "practice_weights": {
    "civil_commercial": 0.62,
    "labor": 0.18,
    "criminal": 0.05,
    "ip": 0.15
  },
  "style_preferences": {
    "answer_style": "conclusion_first",
    "detail_level": "medium",
    "citation_style": "explicit"
  },
  "document_preferences": {
    "common_templates": ["civil_complaint", "reply_letter"],
    "preferred_sections": ["facts", "claims", "legal_basis"]
  },
  "risk_preference": "moderate",
  "confidence": 0.78,
  "updated_from": [
    "manual_edit_history",
    "accepted_answers",
    "rejected_suggestions"
  ]
}
```

### 更新原则

- 单次更新不超过 10%
- 连续三次同方向行为才固化
- 重要变更进入待确认
- 用户可以手动锁定偏好项

这就是"保留贝叶斯精神，但不神化贝叶斯工程"。

## 6. 知识治理体系

## 6.1 法律知识库不是纯向量库

法律知识必须带元数据。

### 每条知识建议包含

- 标题
- 来源机构
- 来源类型
- 生效状态
- 生效时间/废止时间
- 法域/地域
- 权威等级
- 适用场景
- 抓取时间
- 原文链接
- 向量索引
- 关键词索引

## 6.2 知识分层

```text
法律知识库
├─ 一级：权威规范
│  ├─ 法律
│  ├─ 行政法规
│  ├─ 司法解释
│  └─ 地方规范
│
├─ 二级：裁判资料
│  ├─ 裁判文书
│  ├─ 典型案例
│  └─ 裁判要旨
│
├─ 三级：业务材料
│  ├─ 合同模板
│  ├─ 文书模板
│  ├─ 审查清单
│  └─ 内部 SOP
│
└─ 四级：辅助知识
   ├─ 研究文章
   ├─ 培训材料
   └─ 律所经验总结
```

### 为什么要分层

因为法律 AI 的首要风险就是"说得像真的，但依据不对"。NCSC 的法律 AI 指引把伪造引用、混淆法域、错引程序规则列为核心风险。[^4]

## 7. 风险控制层

## 7.1 输出前必须过四关

### 第一关：来源关

- 结论是否绑定来源
- 引用是否真实存在
- 法域是否匹配
- 是否存在失效条文

### 第二关：风险关

风险分级：

- 低风险：法规解释、一般信息整理
- 中风险：类案分析、合同条款建议
- 高风险：诉讼策略、对外文书、期限判断

### 第三关：权限关

- 是否允许导出
- 是否允许共享群聊
- 是否允许生成正式版
- 是否需要二次确认

### 第四关：人工关

以下任务必须律师确认：

- 起诉状/答辩状/代理词导出
- 合同审查意见形成正式稿
- 对外法律意见书
- 任何带有结论性法律建议的输出

---

## 7.2 标准输出结构

每次回答固定为：

1. 结论摘要
2. 法律依据
3. 类案/条款参考
4. 风险提示
5. 置信度
6. 免责声明

### 示例骨架

```text
【结论】
...

【依据】
1. ...
2. ...

【相关参考】
...

【风险提示】
本结论涉及...，建议结合具体案情复核。

【置信度】
中

【免责声明】
本内容由AI辅助生成，仅供内部工作参考，不构成正式法律意见。
```

## 8. 可观测与评估层

这层是 V3.0 的真正核心。

## 8.1 全链路 Trace

每次任务记录：

- 用户输入
- 意图识别结果
- 风险等级
- 选了哪些工具
- 每个工具返回什么
- 引用了哪些知识源
- 最终输出是什么
- 用户是否接受/修改/驳回

## 8.2 关键指标

### 质量指标

- 法条引用准确率
- 案例引用准确率
- 法域匹配率
- 检索命中率
- 用户采纳率
- 文书初稿保留率

### 风险指标

- 高风险任务漏标率
- 无来源输出率
- 误引/错引率
- 人工重写率
- 被用户投诉率

### 运行指标

- 平均响应时长
- 平均工具调用次数
- 单任务 Token 成本
- 检索耗时
- 系统异常率

### 漂移指标

- 用户偏好突变率
- 知识失效命中率
- 检索相关度下降趋势
- 高风险输出占比变化

## 8.3 评估机制

- 离线 golden set：法规问答、案例检索、文书草拟标准题库[<sup>2][</sup>3]
- 在线 eval：真实流量抽样复核
- 周报：质量、风险、成本、采纳率
- 回滚：提示词、工具路由、偏好快照、知识库版本均可回滚

## 11. V3.0 的最终判断

这版架构的本质不是"更复杂"，而是把复杂性收拢在少数基础设施层里，把用户看到的界面做得更简单。  
真正的先进，不是术语多，而是：

- 出错时能知道哪里错
- 漂移时能知道为什么漂
- 出结果时能知道依据是什么
- 高风险时能把人牢牢拉回来

这才是法律智能体该有的样子。[<sup>2][</sup>3][^4]
